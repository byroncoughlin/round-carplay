The problem

A phone screen did not belong here

The R75/6 is round glass, needles, heat, and vibration. A flat rectangle bolted to the dash would fix navigation and wreck the look of the cockpit.

So I set one rule. CarPlay had to fit the bike, not take it over.

f-io / pi-carplay got it started. It proved a Raspberry Pi and a Carlinkit adapter can run CarPlay.

The circle

Fit CarPlay inside a circle

The first motoCarPlay build came from OneMakerShow / round-carplay. The idea is simple: put CarPlay in the largest square that fits an 800×800 round screen.

That leaves four curved corners. Instead of wasting them, I turned them into the gauges: speed, heading, cylinder heat, lean, altitude, and air temp.

That prototype is the demo on this page. It nailed the layout before the hardware was ready.

The rebuild

From demo shell to a real stack

The layout worked, but the Pi needed more than a demo shell. I moved it onto LIVI for a native, stable display stack.

LIVI adds GStreamer video, hardware decode, an embedded compositor, and a cleaner head-unit design. Same idea, better plumbing.

No data bus

Old bike, no data port

A 1975 airhead has no OBD port. Every number on screen comes from a sensor I added, read by the Pi, and placed where an old gauge would sit.

GPS came first. It gives speed, heading, and altitude without touching a wheel, the gearbox, or a cable.

Adafruit Ultimate GPSspeed · heading · altitude

Air · silicon

Outside air, and the Pi itself

A waterproof probe reads the outside air. The Pi reports its own CPU temperature from inside the case.

One number's about the ride. The other tells me whether the computer buried in the dash is cooking.

DS18B20 probePi CPU temp

Airhead heat

Watch both cylinders breathe

This engine is air-cooled, just wind over the fins. A K-type thermocouple under each spark plug reads the left and right heads, where the heat actually matters.

The gauges go blue when cold, amber when warm, red when too hot. You read the color before the number.

Each probe has its own amplifier and shares an SPI bus back to the Pi.

2× MAX31856K-type probe · 14 mm

Motion · IMU

The dash should lean with the ride

A 9-axis sensor tracks lean, pitch, and G-force live. Lean into a corner and the display leans with you.

It's not a race dash. It just shows what the bike's actually doing.

BNO055 9-DOFlean · pitch · G

Memory

Tap a gauge for the history

The live readout is only half of it. Every value is logged while the app runs, so a tap opens a graph with the current reading, min and max, and the recent trace.

Temperature graphs add color bands behind the line, so a hot trend is something you see instead of a number you babysit.

The disguise

Let CarPlay color the whole gauge

The last problem was the square. Black corners made CarPlay look like a screen dropped into a hole.

An optional backdrop samples the live frame, scales it up, blurs it, and bleeds that color into the round edges. The square's still there, it just stops fighting the circle.

Where it landed

What I run now is LIVI

This page is the story and the simulation. The code I run on the bike is the LIVI fork: hardware-accelerated video, clean mode switches, rounded CarPlay corners, and the backdrop as an opt-in.

This was a relay, so the credits stay up. pi-carplay opened the door, round-carplay shaped the display, and LIVI made the Pi build solid. motoCarPlay is my bike's version of all of it.
